# 推荐工作流与输出清单

本文件规定如何从用户资料卡、政策规则和数据脚本生成“冲、稳、保、垫、兜底”院校专业推荐清单。

## 总流程

1. 读取用户资料卡。
2. 读取 `<skill_dir>/references/2026-policy-rules.md`。
3. 读取 `<skill_dir>/references/data-sources.md`。
4. 使用 `<skill_dir>/scripts/equivalent_score.py` 获取用户分数/位次在 2022-2025 的等位参考。
5. 使用 `<skill_dir>/scripts/query_admissions.py` 查询院校专业组、专业录取分、招生计划、章程链接。
6. 将候选项按用户偏好、风险层级、政策限制、经济限制过滤。
7. 按目标批次官方容量设置覆盖率目标，并尽量扩展到足够多的合适候选项。
8. 生成结构化推荐清单。
9. 等用户确认后，把最终结构化推荐数据保存为 `output/{考生代号}-{科类}-{位次}/recommendations-confirmed.json`。
10. 后续 Excel 渲染交给 `<skill_dir>/references/final-form-guide.md`；最终交叉验证交给 `<skill_dir>/references/final-cross-validation-guide.md`。

## 信息不足时

如果缺少位次、科类、目标批次、是否艺体等核心信息，先向用户追问。不要用模糊信息直接生成正式推荐。

如果缺少兴趣、地域、经济、职业规划等软信息，可以先做“数据候选池”，但正式推荐前必须提醒用户补充偏好。

## 候选池构建

普通类本科/专科推荐优先使用通用批量筛选脚本，不要为每个考生临时重写候选池脚本：

```bash
python <skill_dir>/scripts/build_recommendations.py --candidate output/{考生代号}-{科类}-{位次}/candidate.json --output-dir output/{考生代号}-{科类}-{位次}
```

该脚本会一次性读取 `school-scores`、`major-scores`、`plans`、`charters`，输出：

- `output/{考生代号}-{科类}-{位次}/recommendations-review.md`
- `output/{考生代号}-{科类}-{位次}/intermediate/recommendations-proposed.json`
- `output/{考生代号}-{科类}-{位次}/intermediate/candidate-pool.json`
- `output/{考生代号}-{科类}-{位次}/query-log.md`

艺术、体育、春考、学考、3+证书和其他特殊类别目前只提供数据查询辅助，不使用 `build_recommendations.py` 端到端自动生成推荐清单。遇到这些类别时，先用 `query_admissions.py` 查询对应 source，再按 `<skill_dir>/references/2026-policy-rules.md` 和官方最新目录人工复核。若普通类确需额外批量逻辑，应优先扩展 `<skill_dir>/scripts/build_recommendations.py`，不要在 `output/` 下长期保留一次性筛选脚本作为主流程依赖。

普通类建议先使用院校专业组层数据：

```bash
python <skill_dir>/scripts/query_admissions.py --source school-scores --year 2025 --subject 物理类 --batch 本科批 --rank-center 50000 --rank-width 20000 --limit 80
```

再用专业分和招生计划交叉：

```bash
python <skill_dir>/scripts/query_admissions.py --source major-scores --year 2025 --subject 物理类 --school-exact 中山大学 --limit 50
python <skill_dir>/scripts/query_admissions.py --source plans --year 2025 --subject 物理类 --school-exact 中山大学 --limit 50
```

如果用户给专业偏好，先按专业关键词查专业层数据，再回到专业组层做风险判断：

```bash
python <skill_dir>/scripts/query_admissions.py --source major-scores --year 2025 --subject 物理类 --major-contains 计算机 --rank-center 50000 --rank-width 30000
```

如果查询具体院校，优先使用 `--school-code` 或 `--school-exact`。只有做模糊探索时才使用 `--school-contains`。

如果 `major-scores` 对目标院校返回 `No rows matched`，不要假设该校没有相关专业。改用 `plans` 查询专业代码、专业组、计划数和备注，并在推荐清单中标记“专业分缺失，已用招生计划补查”。

招生章程查询：

```bash
python <skill_dir>/scripts/query_admissions.py --source charters --school-exact 清华大学
```

艺术/体育类先用对应专区数据，再按 2026 合成分规则和最新专业目录复核。

## 覆盖率目标

推荐清单默认不是“挑几个最喜欢的学校”就结束，而是在符合资料卡的前提下尽量覆盖目标批次容量。这样做的原因是广东普通类平行志愿给了较多院校专业组志愿位，多填入可接受且梯度合理的志愿，通常能提升录取机会并降低滑档风险。

按目标批次设置覆盖目标：

- 普通类本科院校：目标 45 个完整志愿。
- 普通类专科院校：目标 45 个完整志愿。
- 军检院校、教师专项、卫生专项、定向培养军士等：目标按官方容量执行。
- 艺体类本科/专科：目标按官方容量执行，并额外复核统考、校考、合成分和兼报限制。

推荐阶段应先构建大于目标容量的候选池，再筛选成最终清单。普通类本科/专科建议候选池至少覆盖目标容量的 1.5 倍，便于剔除专业组调剂风险、费用风险、地域不匹配和代码待核查项。

如果初次查询后合适候选不足，不要直接停在少量志愿。按以下顺序扩展：

- 扩大位次窗口，例如从 `--rank-width 20000` 扩展到 `30000`、`50000`，并保留冲稳保垫兜底梯度。
- 放宽专业关键词到相邻专业方向，例如“计算机”扩展到软件工程、网络空间安全、数据科学、人工智能、电子信息、自动化等用户可接受方向。
- 放宽地域到用户可接受的相邻城市、省内其他城市或明确可接受的外省区域。
- 增加同层次但调剂风险更低、收费更可控的院校专业组。
- 对保、垫、兜底层优先补充公办、计划稳定、专业组内可接受专业较多的院校专业组。

只有满足以下任一条件，才允许最终清单少于官方容量：

- 用户资料卡明确强偏好，且用户确认不愿为了填满而降低院校、专业、地域或费用匹配度。
- 考生分数/位次极高，采用强匹配策略时，继续补充会明显降低意向度或引入不必要风险。
- 用户明确排斥范围很大，例如不接受外省、不接受高收费、不接受调剂、只接受少量专业方向。
- 当前数据源检索后合适院校专业组不足，且已在 `query-log.md` 或推荐清单中记录查询条件和无结果情况。
- 目标批次本身容量较小，或特殊类别资格、体检、性别、语种、统考类别等政策条件限制了可选范围。

未满额时，推荐清单必须包含：

- 完整志愿数/官方容量。
- 未满额例外类型：`strong_preference`、`high_score_strategy`、`hard_constraints`、`data_insufficient`、`policy_limited` 或 `user_requested_sparse`。
- 已尝试的扩展查询和筛选动作。
- 继续补满会损害哪些资料卡目标或引入哪些风险。
- 建议用户可放宽的约束，以及放宽后可补充的方向。

对类似“普通类本科、位次约 50000、无极端强偏好”的普通考生，12/45 这种覆盖率通常不应作为最终推荐结果；应回退扩展候选池，优先补齐稳、保、垫、兜底层的可接受志愿。

## 风险层级

将候选项分为：

- 冲：历史最低位次明显优于用户位次，或等位分高于用户 3-20 分，录取不确定但有兴趣价值。
- 稳：历史最低位次接近用户位次，近年波动可控，专业组内可接受专业较多。
- 保：历史最低位次低于用户位次，计划稳定，退档风险可控。
- 垫：明显低于用户位次，用于降低滑档风险。
- 兜底：足够低且用户可接受；优先公办、专业可接受、调剂风险低。

不要机械套固定分差。必须结合：

- 2025 位次
- 2024/2023/2022 趋势
- 招生计划增减
- 专业组内专业构成
- 用户是否服从调剂
- 高收费/体检/单科/语种限制

## 推荐清单字段

正式清单的结构化 JSON 必须使用 `build_recommendations.py` 的统一 schema，且与 `render_volunteer_draft.py` 可直接对接。每条记录至少包含：

- 志愿号
- 层级：冲/稳/保/垫/兜底
- 录取批次
- 科类
- 院校代码
- 院校名称
- 院校专业组代码
- 专业代码列表、专业代码1-6
- 是否服从调剂
- 调剂条件
- 推荐专业或专业方向
- 组内可接受专业
- 2025专业最低分/位次
- 2025组最低分/位次
- 2024/2023/2022参考
- 招生计划变化
- 地域
- 学校性质
- 标签
- 学费风险
- 推荐理由
- 风险提示
- 章程核查状态
- 代码复核状态
- 备注

`录取批次`、`志愿号`、`院校代码`、`院校名称`、`院校专业组代码`、`专业代码1-6`、`是否服从调剂`、`层级`、`备注` 是最终 Excel 的直接渲染字段，不得改名。历史专业组代码和专业代码必须在 `代码复核状态` 与 `备注` 中标记“待2026官方招生专业目录复核”，不得冒充 2026 最终代码。

`recommendations-review.md` 必须包含目标批次覆盖情况：完整推荐数/官方容量。未满额时还必须写明未满额例外类型、扩展查询记录和可放宽约束建议；如果已满额，写明“已达到官方容量”。

## 反模式

不要这样做：

- 只按分数推荐，不看位次。
- 只按院校名称推荐，不看专业组。
- 只看目标专业，不看同组调剂专业。
- 把用户明确排斥的地区、专业、收费类型放入保底。
- 对大多数普通考生只给少量志愿后直接进入填表；除非有强偏好、高分强匹配、硬约束或数据不足证据，否则应继续扩展候选池。
- 使用 2022-2025 历史数据冒充 2026 最终招生计划。
- 忽略体检、单科、语种、性别、色盲色弱等章程条件。
- 未经用户确认就直接生成最终志愿表。

## 用户确认前输出

先输出“推荐清单供审阅”，保存为项目根目录 `output/{考生代号}-{科类}-{位次}/recommendations-review.md`。不要直接填最终志愿表。请用户确认：

- 是否接受每个层级的风险比例。
- 是否删除任何院校/专业。
- 是否加入新的院校、地区或专业偏好。
- 是否接受高收费、中外合作、联合培养等项目。
- 是否接受服从调剂。
- 如果当前推荐数少于官方容量，是否愿意放宽专业、地域、院校层次、收费或调剂约束以继续补充志愿。

用户确认后，从 `intermediate/recommendations-proposed.json` 中保留用户确认的条目，顺序调整后重写 `志愿号`，并保存为 `output/{考生代号}-{科类}-{位次}/recommendations-confirmed.json`。确认文件必须保留上述统一 schema；不要只保存 Markdown 表格，也不要把字段改成最终渲染脚本无法识别的别名。

## 交接给最终步骤

用户确认后，本文件职责结束。不要在本文件继续执行最终交叉验证；改按以下顺序交接：

1. 读取 `<skill_dir>/references/final-form-guide.md` 生成 `volunteer-draft.xlsx`。
2. 读取 `<skill_dir>/references/final-cross-validation-guide.md` 生成 `validation-report.md`。
3. 验证通过后读取 `<skill_dir>/references/completion-summary-guide.md` 生成 `completion-summary.md`。
